# Qitus SaaS Context

## Domain Vocabulary

- **Company**: the single legal entity managed by one authenticated user in the MVP.
- **CompanyWorkspace**: the authenticated working context for one request: User, active Company, active FiscalYear, primary BankAccount, and access/subscription state.
- **FiscalYear**: the active accounting period for a Company.
- **FiscalYearCenter**: the Module that lists, creates, validates and activates FiscalYears for one mono-company workspace.
- **ActiveFiscalYear**: the FiscalYear selected by query/cookie for the current request; all accounting Modules read it through CompanyWorkspace.
- **Import**: one uploaded bank CSV and its processing lifecycle.
- **ImportOrchestrator**: the Module that owns CSV import outcomes: file storage, detection, parsing, mapping continuation, categorization, ledger writing, retries, status, and activity logging.
- **ImportExecutor**: the Seam behind ImportOrchestrator for execution mode; `InlineImportExecutor` runs immediately, `BullMqImportExecutor` queues the same work for an import worker.
- **DemoDatasetSeeder**: the Module that rebuilds local demo datasets from fixtures, keeping `qonto_mvp` reproducible while exposing hardening datasets for multi-bank CSV, real VAT, and beta closing evidence.
- **Transaction**: one normalized bank movement extracted from an Import.
- **TransactionExplorer**: the Module that owns transaction listing, filters, pagination, facets, display account, and business status.
- **TransactionFilterState**: the Module that owns transaction URL filters, normalization, serialization, and readable active-filter labels.
- **TransactionReviewQueue**: the Module that owns the user review queue for transactions whose categorization is `NEEDS_REVIEW`, including next/previous navigation.
- **TransactionFacets**: the aggregate counts returned with a transaction search, such as review, corrected, rule-backed, debit, and credit counts.
- **TransactionSuggestion**: a deterministic candidate categorization for one Transaction, sourced from current categorization, CorrectionRule, VendorMapping, pattern, or review fallback.
- **Categorization**: the accounting account proposal or confirmation for a Transaction.
- **CorrectionRule**: a Company-specific learned mapping applied before global mappings and IA.
- **CorrectionRuleCenter**: the Module that owns learned correction rules: creation, activation lifecycle, previewed impact, and activity logging.
- **CorrectionRuleImpact**: the business reading of a CorrectionRule impact: matched transactions, examples, conflicts, and warnings for broad rules.
- **VendorMapping**: a deterministic global or Company mapping from vendor/label patterns to PCG accounts.
- **JournalEntry**: a balanced double-entry accounting entry generated from confirmed Categorizations.
- **JournalExplorer**: the Module that owns journal listing, filters, pagination, facets, totals, and debit/credit balance.
- **JournalExport**: a stable export of JournalEntries, such as CSV or JSON, generated from `JournalExplorer` results.
- **JournalBalance**: the business check that total debit equals total credit for a journal view or FiscalYear.
- **JournalAudit**: the proof-oriented reading of a FiscalYear journal, detecting imbalance, empty entries, invalid lines, and unexpected sources before export.
- **VatLedgerPolicy**: the Module that decides whether TVA applies to a Transaction and builds HT/TVA/TTC JournalLines for real VAT regimes.
- **VatOperationNature**: the deterministic tax nature attached to a categorization or rule, such as domestic purchase, domestic sale, reverse charge, exempt, or out of scope.
- **VatExigibility**: the Company-level timing rule used by declarative TVA reads: encaissements, débits, or mixed.
- **VatPosition**: the Module-owned declaration view of one period, aggregating bases, deductible TVA, collected TVA, reverse-charge TVA, and TVA account balances.
- **VatDeclaration**: a local CA3 or CA12 draft persisted as structured JSON plus a downloadable markdown Document; it is not teletransmitted.
- **VatControl**: one deterministic TVA check such as missing rate, missing nature, stale declaration, or open TVA account balance.
- **VatRegularization**: the read-only preview of TVA to pay or credit to carry forward; Phase 12 does not create settlement entries automatically.
- **VatRatePolicy**: the Module that owns supported TVA rates, operation natures, labels, parsing, and compatibility checks.
- **VatReviewWorkflow**: the guided workflow that turns VatControls into stable VAT issues that can be resolved through existing transaction correction or declaration regeneration Modules.
- **VatDeclarationFreshness**: the calculated lifecycle state of a VAT declaration draft: active, stale, or superseded, based on later imports, corrections, entries, OD, or fiscal profile changes.
- **VatLedgerReadiness**: the calculated state explaining whether existing JournalEntries can produce a usable TVA position after fiscal profile or import changes.
- **ChangeImpact**: the calculated product diagnostic for derived states that became stale or unsafe after a source change, such as documents, VAT drafts, FEC, reconciliations, closing adjustments, evidence, connector sync or expert dossier snapshots.
- **ChangeImpactCenter**: the Module that aggregates ChangeImpacts from domain-owned freshness/readiness Modules without becoming the accounting source of truth.
- **Document**: an artifact generated by Qitus scripts, such as FEC or statements.
- **DocumentCatalog**: the Module that owns document listing, detail, download metadata, and freshness-enriched document rows.
- **DocumentStorageAdapter**: the Seam for storing generated document bytes; local filesystem is the default Adapter in the MVP.
- **DocumentGenerationAudit**: the ActivityLog-backed proof of document generation attempts, successes, failures, scriptVersion, filenames, duration, and user-facing messages.
- **DocumentEvidenceBundle**: the Module that assembles local proof material for one FiscalYear: generated documents, journal CSV, and a manifest.
- **EvidenceManifest**: the JSON evidence index listing company, FiscalYear, journal audit summary, document metadata, scriptVersion, and generation timestamps.
- **Attachment**: an uploaded accounting proof file for one Company/FiscalYear, stored locally in beta with optional extracted text and invoice metadata.
- **AttachmentLink**: the auditable relation between an Attachment and a Transaction, JournalEntry, ClosingAdjustment, or FiscalYear.
- **AttachmentExtraction**: the non-blocking OCR/text extraction state for one Attachment; failure keeps the file usable through manual metadata.
- **EvidenceStorage**: the Seam for storing uploaded proof bytes; local filesystem is the default Adapter in Phase 11.
- **EvidenceControl**: the Module-owned review of evidence coverage, including entries without proof, orphan attachments, and OCR failures.
- **EvidenceBundleAttachment**: an Attachment referenced in the evidence bundle manifest with availability, hash, metadata, and accounting links.
- **EvidenceReviewWorkflow**: the guided workflow that turns missing EvidenceRequirements into resolved AttachmentLinks through upload-from-requirement or explicit link-to-requirement actions.
- **AttachmentMatching**: the deterministic scoring Module that suggests AttachmentLinks from amount, date, supplier text, and expected proof type, without auto-linking outside a requirement upload.
- **EvidenceAudit**: the Module-owned audit of proof links and local files, detecting broken AttachmentLinks and missing stored evidence bytes before building the bundle.
- **ResolvedEvidenceRequirement**: an EvidenceRequirement satisfied by a compatible AttachmentLink or expert validation, removed from the active proof review queue but kept visible for audit.
- **AnnualClosing**: the guided 12-step FiscalYear closing workflow that turns pre-closing checks, OD, documents, and evidence into an auditable close decision.
- **AnnualClosingRun**: the persisted closing state for one FiscalYear, including status, timestamps, final evidence manifest, and the ordered AnnualClosingSteps.
- **AnnualClosingStep**: one deterministic or user-reviewed step in the annual closing workflow, with blockers, warnings, evidence, and completion state.
- **FixedAssetRegister**: the Module that owns fixed assets and deterministic depreciation previews for closing.
- **BankReconciliation**: the Module-owned comparison between ledger cash account balance and the bank statement balance at FiscalYear end.
- **ReconciliationRun**: one line-based reconciliation pass for a FiscalYear, scoped to bank, Stripe, third-party matching, or suspense accounts.
- **ReconciliationMatch**: one auditable line match between two business objects, such as a bank Transaction and a bank JournalLine, with status, confidence and differences.
- **ReconciliationIssue**: one stable point found by a ReconciliationRun that must be resolved, ignored with note, or carried as a warning.
- **ReconciliationFreshness**: the calculated state telling whether a ReconciliationRun is up to date or must be rerun after imports, corrections, journal changes, OD approvals, Stripe data or bank statement balance changes.
- **ReconciliationReview**: the guided queue of ReconciliationIssues across bank, Stripe, third-party and suspense accounts, with note-backed resolve, ignore and reopen decisions.
- **ReconciliationReport**: the evidence JSON produced from ReconciliationRuns, matches, issues, freshness, user notes and statistics for the evidence bundle.
- **BankLineReconciliation**: the Module-owned line-by-line match between imported bank Transactions and ledger lines on the bank account.
- **StripeReconciliation**: the Module-owned match between Stripe payouts/events and bank/journal evidence, including fees, refunds and disputes as review points.
- **ThirdPartyMatching**: the Module-owned lettrage of customer, supplier and other third-party accounts.
- **SuspenseAccountReview**: the Module-owned review of suspense accounts `471`, `467`, `511` and `580`.
- **ConnectorSync**: the Module that runs disabled, fixture or live Qonto/Stripe sync through Adapters without storing provider secrets in the database.
- **ConnectorStatus**: a non-secret diagnostic view of connector mode, source and configuration readiness for Qonto and Stripe.
- **TaxPackageDraft**: a local `.md` draft of the fiscal package generated from deterministic journal totals and company metadata.
- **TaxPackageTemplateRenderer**: the Module that renders the structured fiscal package source with case labels, amounts, and calculation references.
- **DocumentPdfRenderer**: the Adapter that optionally turns structured fiscal package sources into PDFs via Qitus CLI scripts and Puppeteer.
- **ClosedFiscalYear**: a FiscalYear with status `CLOSED`; business mutations require explicit reopening before they can proceed.
- **AccountingReview**: the pre-closing business reading of one FiscalYear, returning readiness, blockers, warnings, actions, and evidence before document generation.
- **AccountingControl**: one deterministic check inside an AccountingReview, either blocking document generation or warning about a pre-closing point to review.
- **AccountingIssue**: one stable, trackable item produced by an AccountingControl, usually tied to a transaction, account, or FiscalYear.
- **AccountingIssueResolution**: the user-owned follow-up state for an AccountingIssue: open, resolved, ignored, and optional note. It does not create accounting entries.
- **PreClosingIssue**: a non-blocking AccountingControl that points to year-end work such as CCA, depreciation, VAT threshold monitoring, or reconciliation.
- **ClosingAdjustmentProposal**: a deterministic OD proposal linked to an AccountingIssue, stored as calculation JSON and draft journal lines until the user validates or rejects it.
- **ClosingAdjustmentAssumptions**: the editable inputs used to calculate one draft ClosingAdjustmentProposal, such as CCA period, depreciation duration, or corporate tax rate.
- **ClosingAdjustmentCalculation**: the deterministic explanation and amounts produced from ClosingAdjustmentAssumptions.
- **ClosingAdjustmentEvent**: an internal audit event attached to one ClosingAdjustmentProposal, tracking assumption changes, recalculation, approval, and rejection.
- **ClosingAdjustmentCenter**: the Module that creates, previews, validates, rejects, and summarizes ClosingAdjustmentProposals.
- **ClosingAdjustmentJournalEntry**: a real `OD` JournalEntry created only after user approval of a ClosingAdjustmentProposal.
- **ClosingWorkpaper**: a user-owned closing workpaper carrying assumptions, notes, optional source entity, and proof expectations before any OD proposal is validated.
- **ClosingWorkpaperWorkflow**: the guided review Module for ClosingWorkpapers, exposing draft/ready/archive state, expected proposal, missing evidence and next workpaper navigation.
- **GeneralClosingAdjustment**: a deterministic draft OD generated from a ClosingWorkpaper or reconciliation/TVA issue, still requiring explicit user approval.
- **ClosingAdjustmentReview**: the guided review state for one ClosingAdjustmentProposal, combining calculation, freshness, evidence gate and approval/rejection actions.
- **ClosingAdjustmentFreshness**: the calculated state telling whether a draft ClosingAdjustmentProposal must be recalculated after workpaper, evidence, import, transaction, journal, OD, TVA, reconciliation or fiscal profile changes.
- **StaleClosingAdjustment**: a draft ClosingAdjustmentProposal whose stored calculation is older than a relevant business event and cannot be approved until recalculated.
- **ClosingAdjustmentEvidenceGate**: the proof requirement that blocks approval of OD kinds requiring a compatible AttachmentLink, without blocking draft generation.
- **AccrualClosing**: the closing domain for FNP, FAE, PCA and generalized CCA proposals.
- **InventoryClosing**: the closing domain that calculates stock variation proposals from initial and final stock assumptions.
- **ProvisionClosing**: the closing domain for provisions and provision reversals documented by workpapers.
- **LoanClosing**: the closing domain that calculates prorata temporis accrued interest proposals.
- **PayrollClosing**: the closing domain for payroll and social charge accrual proposals.
- **TaxClosing**: the closing domain for local tax calculation workpapers and IS proposals.
- **ReconciliationAdjustment**: the closing domain that can transform auditable reconciliation issues into OD proposals, never direct journal entries.
- **VatSettlementAdjustment**: the closing domain that consumes VatRegularizationCenter to propose TVA settlement OD without payment automation.
- **DocumentFreshness**: the calculated state that tells whether generated Documents include the latest imports, corrections, JournalEntries, and validated OD.
- **StaleDocument**: a Document that was generated before a newer business event and should be regenerated.
- **WebhookEvent**: one external provider event received by Qitus SaaS and processed idempotently.
- **ActivityTimeline**: the user-facing reading of ActivityLog rows, with French labels and compact metadata.
- **AuditExport**: the ActivityLog-backed audit export for one CompanyWorkspace, available as CSV or JSON with filters.
- **AccountingCoverage**: the calculated expert-comptable readiness view of one FiscalYear, summarizing covered, partial, missing, and high-risk areas.
- **CoverageArea**: one AccountingCoverage domain such as TVA, FEC, liasse, evidence, reconciliations, closing, or expert review.
- **EvidenceRequirement**: a required or recommended proof expected for a JournalEntry, Transaction, ClosingAdjustment, or FiscalYear.
- **MissingEvidence**: an EvidenceRequirement not yet satisfied by a compatible AttachmentLink or expert validation.
- **BetaReadiness**: the product-facing readiness surface that aggregates dossier coverage plus runtime, dependencies, storage, workers, Open Banking, webhooks, secrets and validation checks before beta; it is not a legal certification.
- **Notification**: one persisted product alert derived from business Modules, with read/dismiss state and a stable dedupe key.
- **NotificationCenter**: the Module that refreshes, lists, summarizes, reads and dismisses Notifications without making them a new source of business truth.
- **RegulatoryFreshness**: the local freshness state of regulatory data checks, currently activity-backed until Phase 11 cron jobs.
- **RegulatorySourceSnapshot**: one immutable retrieval of an official source such as BOFiP RSS, ANC/PCG or impots.gouv documentation, identified by checksum and source URL.
- **RegulatoryChange**: one detected change derived from a RegulatorySourceSnapshot, kept as traceable official evidence before Qitus turns it into executable rules.
- **AccountingRulePack**: a versioned set of deterministic Qitus accounting rules, activated for future imports while older packs are archived.
- **RuleImpactPreview**: the calculated preview of existing transactions, user CorrectionRules and derived artifacts that would be affected by a newer AccountingRulePack.
- **AutomaticRuleApplication**: the workspace record proving the active AccountingRulePack applies to future imports; existing entries are never rewritten silently.
- **PrivacyRequest**: one user data request such as export, soft delete, anonymization or purge, tracked for RGPD audit.
- **PrivacyCenter**: the Module that handles soft deletion, anonymization and protected purge decisions.
- **DataExportCenter**: the Module that builds the complete user/company JSON export for portability.
- **ExpertReviewShare**: a read-only external review of one Company/FiscalYear dossier through a tokenized ShareLink.
- **ShareLink**: a hashed-token access grant for external read-only review, with expiration, revocation and optional expert validation.
- **ExpertValidation**: the reviewer name, note and timestamp proving that an external expert-comptable reviewed the shared dossier.
- **ExpertDossier**: the complete expert-comptable review dossier for one FiscalYear, aggregating FEC, liasse, journal audit, VAT, evidence, reconciliations, closing workpapers, OD, activity and review state.
- **DossierSnapshot**: a persisted manifest of the dossier state transmitted to an expert; later accounting or evidence changes make it stale.
- **FecPrecheck**: the Module-owned pre-export reading of the FEC, comparing document freshness, format and entry counts against the audited journal.
- **TaxPackageCompletion**: the Module-owned reading of whether the structured liasse source is present and fresh; PDF remains optional.
- **ExpertReviewRun**: one collaborative review cycle for a shared dossier, from draft/in-review to changes requested or final signoff.
- **ExpertReviewItem**: one expert-comptable request, question or blocker tied to a dossier section or business entity.
- **ExpertReviewComment**: one user, expert or system comment attached to an ExpertReviewItem.
- **ExpertSignoff**: the final expert-comptable validation of a review run, allowed only when blocking review items are resolved.
- **ExpertDossierExport**: the local JSON export of the final dossier, including evidence bundle, FEC, liasse, review comments, signoff and activity audit.
- **ExpertDossierReadiness**: the guided reading of what blocks or warns before sharing a dossier or exporting the final cabinet package.
- **SnapshotDiff**: the comparison between a transmitted DossierSnapshot and the current dossier state, including freshness reasons.
- **SharedReviewPortal**: the read-only projection exposed to an external expert through a ShareLink; only comments, requests and signoff are mutable.
- **ExpertReviewQueue**: the owner-side queue of expert requests, answers, waivers and resolutions.
- **ExpertDossierExportVerification**: the Module-owned verification that the exported cabinet package contains the expected artifacts and final signoff.
- **AccountingChat**: the read-only conversational assistant for one CompanyWorkspace; it can explain and point to screens, but it cannot mutate accounting data.
- **ChatContext**: the compact business snapshot supplied to AccountingChat, built from Dashboard, AccountingReview, ClosingAdjustment, JournalAudit, DocumentFreshness and AnnualClosing Modules.
- **ChatReadOnlyPolicy**: the deterministic guard that blocks mutation-like chat requests before they reach an IA provider.
- **ChatAnswerGrounding**: the Module that attaches product references to chat context and replies so answers point back to concrete Qitus screens.
- **ChatReadiness**: the runtime reading of chat provider, model, read-only state and current quota availability.
- **Subscription**: the Company billing state, either local stub or Stripe-backed, carrying tier, status, provider and period metadata.
- **Entitlement**: a product capability checked before use, such as chat or CSV import, based on Subscription tier and UsageMeter counts.
- **EntitlementStatus**: the explicit allow/block state for one capability, including monthly quota and per-minute rate-limit reasons.
- **UsageMeter**: the Module that records and summarizes monthly usage events such as AI chat messages and imports.
- **BillingStatus**: the product-facing billing summary: subscription, usage, entitlements, Stripe readiness and latest webhook processing events.
- **BillingWebhookEvent**: one Stripe billing event received and processed idempotently.
- **DashboardAlert**: a user-facing business alert derived from deep Modules such as AccountingReviewCenter, DocumentFreshnessCenter, ClosingAdjustmentCenter, ImportHistory, and TransactionExplorer.
- **OperationalDashboardConsistency**: the Module that verifies dashboard counters and operational alerts match the transaction, document, OD, and rule Modules.
- **EndUserValidation**: the browser-level MVP validation that proves the demo path through dashboard, corrections, documents, download, and activity works as a user sees it.
- **DocumentRuntimeAdapter**: the module that adapts SaaS data into the filesystem shape expected by the Qitus repo.
- **DocumentExecution**: one structured execution of a Qitus script, including args, timeout, stdout/stderr, exit code, scriptVersion, timestamps, and user-facing error message.
- **DeploymentRuntime**: the production-shaped runtime reading of env configuration, with strict beta/production checks and secret-safe status output.
- **HealthCheck**: the liveness/readiness Module that checks runtime config, database, Redis when required, Qitus scripts, storage and Open Banking provider state.
- **Monitoring**: the local-first metric and error reporting Module used before external Sentry/OTEL adapters are enabled.
- **ObjectStorage**: the S3-compatible storage Adapter used by documents and evidence when `OBJECT_STORAGE_MODE=s3`; local storage remains the dev default.
- **OpenBankingProvider**: the read-only provider Adapter for bank accounts, balances, consent status and transactions; Qitus never implements PSD2 bank-by-bank directly in beta.
- **BridgeOpenBankingAdapter**: the OpenBankingProvider Adapter for Bridge API, using Bridge-hosted consent, non-secret provider ids in Prisma, and bank feed import through BankFeedNormalizer.
- **PowensOpenBankingAdapter**: the OpenBankingProvider Adapter for Powens API, using Powens Webview consent, a vaulted user token, and bank feed import through BankFeedNormalizer.
- **ProviderWebviewConsent**: a provider-hosted consent flow where Qitus redirects the user to Bridge or Powens instead of listing institutions locally.
- **ProviderCredentialVault**: the encrypted local Seam that stores provider user tokens outside Prisma; business Modules only read provider connection ids and never expose raw tokens.
- **EInvoice**: one incoming structured supplier invoice detected from upload or provider sync, preserving source XML, extracted amounts, supplier identity, VAT breakdown and parsing status.
- **StructuredInvoiceParser**: the deterministic parser Seam for Factur-X, UBL and CII; it extracts a canonical invoice payload and never creates accounting entries.
- **EInvoiceAccountingDraft**: the user-reviewable accounting proposal derived from an EInvoice; only approval creates a `JournalEntry` with source `E_INVOICE`.
- **EInvoiceProviderAdapter**: the PA-neutral reception Seam for future Plateforme Agréée integrations; the mock Adapter validates sync without selecting a concrete PA.
- **GenericAccreditedPlatformAdapter**: the live-shaped but non-network Adapter documenting the PA contract until a concrete Plateforme Agréée is selected and implemented.
- **QontoAccreditedPlatformAdapter**: the guarded concrete Adapter for Qonto PA selection; it exposes safe readiness and refuses network calls until Qonto PA sandbox/API documentation and contract tests are validated.
- **QontoPaReadiness**: the diagnostic Module tracking Qonto PA contract, sandbox credentials, webhook secret, formats, statuses, acknowledgement rules, pagination and proof export readiness without exposing secrets.
- **AccreditedPlatformSelection**: the Qitus reading of candidate PA readiness, including documentation, sandbox, webhooks, formats, proof export, pricing and e-reporting scope.
- **AccreditedPlatformSandboxAdapter**: the strict internal PA sandbox Adapter that exercises duplicate, rejected, cancelled, invalid XML and missing visual cases without claiming legal compliance.
- **EInvoiceProviderContractTestKit**: the provider-agnostic contract test Module that validates connect, callback, webhook, sync, payload download, status acknowledgement and disconnect.
- **EInvoiceProviderConnection**: the company-level PA connection state, including non-secret provider ids, mandate status, last sync, last status refresh and safe capabilities.
- **EInvoiceLifecycle**: the Module that translates provider invoice statuses into Qitus invoice states without creating accounting entries.
- **EInvoiceAuditTrail**: the reviewable trace of PA webhooks, syncs, stored XML, provider status changes, drafts and accounting approval.
- **EInvoiceMatching**: the calculated matching between an EInvoice and existing transactions or journal entries, based on amount, date, supplier and invoice reference.
- **BankConsent**: the provider-backed user consent state, including expiration, revocation and reconnect needs without storing raw bank credentials.
- **BankFeedNormalizer**: the Module that converts provider account and transaction payloads into canonical bank movements ingested through `ImportOrchestrator`.
- **BankFeedSync**: one Open Banking synchronization attempt, with fetched/imported counts, user-readable errors and activity audit.
- **SecurityHardening**: beta runtime checks for secure cookies, session secret, signed webhooks, secret redaction and closed fiscal-year mutation guards.
- **OpenBankingSyncWorkflow**: the guided Module for connection detail, sync history, connection-scoped sync and mock reconnect actions.
- **OpenBankingFreshness**: the calculated state telling whether a bank consent is expired, never synced, fresh or stale after imports, corrections or bank reconciliation changes.
- **OpenBankingWebhook**: the idempotent WebhookEvent-backed receiver for Open Banking provider events, with signature verification and no provider secrets exposed.
- **StorageAudit**: the Module-owned audit of generated documents and evidence files across local or S3 storage, reporting missing artifacts without throwing user-facing application errors.
- **WorkerRuntime**: the non-secret runtime reading of import workers and cron mode, independent from a live worker process.
- **MetricCatalog**: the stable list of beta metrics expected for imports, Open Banking, documents, chat, billing, closing and deterministic hit rate.
- **BetaOpsRunbook**: the operational checklist for starting web/worker, checking health, running migrations, validating Open Banking mock, switching storage and rolling back locally.

## Architecture Vocabulary

Use the `improve-codebase-architecture` skill language: Module, Interface, Implementation, Depth, Seam, Adapter, Leverage, Locality.

The MVP favors deep Modules. A caller should ask for business outcomes, not orchestrate parser internals, prompt retries, journal numbering, or Qitus workdir details.
